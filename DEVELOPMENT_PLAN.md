# 反重力飞行对战游戏开发步骤与开发清单

## 1. 项目目标重述

基于 `prd.md`，首个可交付版本应实现一套不依赖重型物理引擎的 3D 反重力飞行核心体验，并把它扩展为可验证对战玩法的游戏原型。核心目标有四个：

1. 做出稳定、可调参的反重力飞行运动学系统。
2. 做出有“丝滑感”和“漂浮感”的飞行操控。
3. 做出即使在高空或纯黑背景下也能感知速度的视觉反馈。
4. 在此基础上接入最小对战闭环，验证游戏可玩性。

当前项目路径进一步明确为：

1. 产品类型：低门槛、偏竞技街机的反重力飞行对战游戏。
2. 控制语义：默认持续前进，`W / S` 用于调速度区间或油门意图，而不是“按下才前进”。
3. 对战节奏：以中高速追逐和选线缠斗为主，不鼓励长期低速或悬停。
4. 数据目标：后续能力评估优先关注 `决策 > 反应 > 专注`，因此必须降低操控复杂度带来的测量噪声。

## 2. 开发原则

1. 先验证“飞行手感”，再扩展“对战复杂度”。
2. 物理、视觉、战斗逻辑解耦，避免后期难以调参。
3. 所有关键体验参数都必须配置化，包括 `k`、`mass`、巡航速度、最大速度、FOV 范围、滚转补偿强度、漂移阈值。
4. 每个阶段结束都必须有可运行版本和验收项，不能只停留在代码完成。
5. 默认巡航速度是核心体验的一部分，不把“持续按住前进键”作为操作负担交给玩家。
6. 单人训练、AI 测试和多人对战尽量共用同一套飞行语义，避免能力评估和竞技体验脱节。
7. 设计优先鼓励“选路线、选时机、选风险”，而不是鼓励玩家管理复杂飞控细节。

## 3. 产品路径补充

### 3.1 目标体验

首版游戏应更接近：

- 高速巡航竞技
- 中近距离追逐缠斗
- 强辅助姿态控制
- 清晰可读的速度和威胁反馈

不应更接近：

- 纯飞行模拟
- 可长期静止的悬浮炮台战斗
- 完全依赖复杂油门管理的重学习成本系统

### 3.2 为什么选择这条路径

1. 更容易上手，便于首轮留存和体验验证。
2. 更适合多人对战，控制语义更稳定。
3. 更适合做行为采样，因为玩家表现更能反映决策和反应，而不是飞行器训练程度。
4. 更符合 PRD 中“极致速度感”和“高空环境下解决静止错觉”的目标。

## 4. 推荐开发阶段

### 阶段 A：技术预研与项目骨架

目标：搭起最小可运行框架，验证渲染技术路线、控制语义与调试方式。

执行步骤：

1. 确定技术栈。
   - 推荐：`TypeScript + Vite + Three.js(WebGL)` 先做原型。
   - 原因：WebGPU 可作为第二阶段升级项，先用成熟方案验证手感更稳。
2. 初始化项目结构。
   - 划分 `core/physics`、`core/rendering`、`gameplay`、`ui/debug`、`config`、`telemetry` 模块。
3. 建立基础渲染循环。
   - 实现固定更新步长或半固定步长。
   - 将物理更新与渲染更新分离。
4. 明确控制语义骨架。
   - 从第一版开始引入 `cruiseSpeed` 或 `targetForwardSpeed`。
   - 避免继续沿用“松手即失去所有主动前进意图”的临时逻辑。
5. 建立调试面板。
   - 支持实时调 `k`、`mass`、巡航速度、加速度、FOV、阻力、滚转强度。
6. 建立事件记录接口。
   - 为后续采集反应时间、决策选择、目标切换埋点预留接口。
7. 加入性能采样。
   - 记录单帧物理耗时、渲染耗时、粒子数量、平均 FPS。

交付物：

- 可启动项目
- 基础场景
- 固定 `dt` 更新循环
- 巡航速度/目标速度控制骨架
- 参数调试面板
- 性能监测面板

验收标准：

- 本地可稳定运行
- 支持实时修改关键飞行参数
- 物理更新链路独立于渲染链路

### 阶段 B：飞行器核心运动学系统

目标：完成不依赖物理引擎的飞行状态更新。

执行步骤：

1. 定义飞行器状态结构。
   - `position`
   - `velocity`
   - `acceleration`
   - `orientationQuaternion`
   - `angularVelocity`
   - `targetForwardSpeed`
   - `cruiseSpeed`
2. 实现输入系统。
   - 调速
   - 刹车/减速
   - 偏航
   - 俯仰
   - 横滚
3. 实现巡航速度模型。
   - 飞行器存在默认持续前进速度。
   - `W / S` 作用于目标速度或油门档，而不是直接代表“按下才前进”。
   - 松手后保持当前速度档位，或缓慢回归默认巡航速度。
4. 实现指数平滑或临界阻尼响应。
   - 输入不直接改状态，而是先改目标状态。
   - 由平滑器把目标速度、目标角速度过渡到当前状态。
5. 实现反重力悬浮逻辑。
   - 提供垂直于地表法线的平衡力。
   - 叠加机身姿态导致的推进偏移。
6. 实现基础阻力模型。
   - 前向阻力弱于横向阻力。
   - 防止无上限漂移和失控。
7. 建立参数化质量感。
   - `mass` 影响加速、减速和姿态响应。
   - `k` 影响输入跟手度与阻尼感。
8. 明确低速边界。
   - 不允许长期停在接近静止的状态。
   - 需要保留“可减速”但不鼓励“可悬停”。

交付物：

- 可控制飞行器自由飞行
- 默认巡航与可调目标速度
- 可调平滑响应
- 可调轻重感
- 无物理引擎的状态更新核心

验收标准：

- 停止速度输入后飞行器仍保持默认巡航或合理中低速
- 修改 `k` 或 `mass` 后手感明显变化
- 逻辑计算可控制在目标预算内

### 阶段 C：丝滑操控与漂移手感

目标：把“能飞”提升到“飞起来舒服”。

执行步骤：

1. 实现自动 Banking。
   - 转向输入映射为目标滚转角。
   - 滚转角随速度和转向强度动态变化。
2. 实现向心力补偿。
   - 转弯时给出与滚转状态匹配的辅助横向力。
   - 避免只看起来倾斜但实际转向发飘。
3. 实现惯性漂移。
   - 允许速度向量与朝向短时不一致。
   - 设置最大偏离角阈值和自动回正机制。
4. 实现姿态回正策略。
   - 松开输入后，横滚和平移偏差逐步回归稳定。
5. 做手感调参轮次。
   - 至少建立三组手感预设：轻型、中型、重型。

交付物：

- 自然侧倾动画
- 可控漂移感
- 预设化飞行参数表

验收标准：

- 转向时机身存在自然滚转
- 漂移不会导致玩家完全失控
- 停止输入后姿态能平稳恢复

### 阶段 D：速度感与空间参照系统

目标：解决高空静止错觉，让“快”被明显感知。

执行步骤：

1. 实现动态 FOV。
   - 根据速度标量映射相机 FOV。
   - 设置最小/最大阈值与平滑过渡。
2. 实现空速线/空间尘埃粒子。
   - 粒子围绕相机生成。
   - 根据飞行方向反向运动并循环复用。
3. 实现边缘参照物系统。
   - 视野边缘生成云团、碎片、极简几何块。
   - 保证高速时有横向掠过参照。
4. 实现速度特效层。
   - 首选径向模糊、屏幕拉伸、轻微色差中的一种或两种组合。
5. 建立视觉开关与质量等级。
   - 便于测试性能和定位体验问题。

交付物：

- 动态 FOV 系统
- 粒子速度参照系统
- 近地感/边缘掠过系统
- 高速视觉特效

验收标准：

- 纯黑背景下依然能感知飞行速度
- 加速时 FOV 和速度特效明显变化
- 高空环境不再产生静止错觉

### 阶段 E：最小对战原型

目标：在成熟飞行手感上接入最小可玩的战斗循环。

执行步骤：

1. 确定核心对战模式。
   - 推荐先做 `1v1 / 2v2 竞技原型`。
2. 实现战斗基础规则。
   - 血量
   - 护盾
   - 击毁
   - 重生
   - 倒计时
   - 胜负判定
3. 实现基础武器系统。
   - 主武器：高速射线或弹丸武器
   - 副武器：带锁定或弱追踪导弹
4. 实现目标锁定与准星反馈。
   - 基于视锥和距离判断可锁定目标。
5. 实现受击反馈。
   - 护盾闪烁
   - 命中特效
   - 音效占位
6. 实现简易 AI 敌机或本地双机测试。
   - 优先保证战斗循环可反复验证。
7. 建立基础行为埋点。
   - 记录目标切换、追击时长、开火时机、脱战时机等行为事件。

交付物：

- 可对战的最小场景
- 基础武器系统
- 命中与击毁反馈
- 简单敌机行为或本地测试机制
- 基础行为日志接口

验收标准：

- 玩家能在飞行中完成追逐、瞄准、射击、击毁闭环
- 战斗系统不破坏飞行手感
- UI 能清楚反馈目标、血量、命中和胜负

### 阶段 F：关卡、环境与战斗验证

目标：让飞行与对战在真实场景里成立，并支持能力评估。

执行步骤：

1. 搭建战斗地图原型。
   - 高空浮岛
   - 管道结构
   - 能量环
   - 几何障碍群
2. 实现碰撞近似方案。
   - 不引入重型引擎，使用包围球/AABB/简化网格做命中检测。
3. 加入风险/收益地形。
   - 掩体
   - 狭窄通道
   - 高速穿越区域
4. 验证可读性。
   - 高速状态下，目标、障碍、特效不能互相淹没。
5. 设计标准化训练场。
   - 让玩家在可控环境下完成追击、避障、机会选择等任务。
   - 为能力评估提供低噪声样本场景。

交付物：

- 一张可反复测试的战斗地图
- 简化碰撞系统
- 场景导航参照物
- 一张标准化训练测试场景

验收标准：

- 地图能支撑追逐、规避、绕障、反打
- 高速飞行下仍能识别战场信息

### 阶段 G：性能优化与验收收口

目标：满足 PRD 的性能和体验验收标准，并让评估路径可用。

执行步骤：

1. 优化物理更新。
   - 避免频繁对象创建
   - 复用向量和四元数
   - 缩减每帧数学开销
2. 优化粒子与特效。
   - 使用对象池或 GPU Instancing
   - 按距离和速度动态裁剪
3. 统一调参与预设。
   - 形成可对比的飞行手感配置表。
4. 做验收测试。
   - 性能
   - 手感
   - 速度感
   - 对战可玩性
5. 建立首版能力评估指标。
   - 决策：目标切换、追击/脱战、风险选择
   - 反应：威胁出现后的规避延迟、开火延迟
   - 专注：长时间追踪稳定度、漏检率
6. 输出开发文档。
   - 核心参数说明
   - 架构说明
   - 后续扩展点

交付物：

- 优化后的可玩版本
- 参数说明文档
- 验收测试记录
- 首版行为指标说明

验收标准：

- 逻辑计算时间接近或达到 `1ms` 目标
- 各项 PRD 验收标准可逐条验证

## 7. 每周执行节奏建议

### 第 1 周

- 完成项目初始化、渲染循环、调试面板
- 建立飞行状态结构和基础调速输入控制
- 跑通最小飞行原型

### 第 2 周

- 完成巡航速度、平滑响应、反重力、阻力与滑行
- 完成 Banking、漂移、回正
- 产出第一版“手感可调”飞行模型

### 第 3 周

- 完成动态 FOV、粒子系统、速度特效
- 解决高空静止错觉
- 做第一次体验验收

### 第 4 周

- 接入最小对战系统
- 完成武器、命中、击毁、HUD
- 建立 AI 靶机或双机对战测试
- 建立基础行为日志

### 第 5 周

- 搭建地图原型与障碍布局
- 验证追逐、规避、绕障、交火体验
- 搭建标准化训练场景
- 做性能优化和参数收口

## 8. 风险与规避

1. 风险：过早做复杂对战，飞行手感未定型。
   - 规避：先冻结飞行核心，再接战斗。
2. 风险：继续沿用“按住前进键才前进”的原型逻辑，导致产品体验偏离。
   - 规避：尽早切换到默认巡航与目标速度模型。
3. 风险：视觉特效很强，但无法提升速度感。
   - 规避：优先做参照物系统，其次才是后处理。
4. 风险：漂移和 Banking 太强，导致操作失控。
   - 规避：所有补偿项都做阈值和预设。
5. 风险：性能浪费在粒子和后处理上。
   - 规避：从一开始就做质量档位与对象池。
6. 风险：能力测量被操控复杂度污染，无法说明真实认知差异。
   - 规避：先标准化控制语义，再设计采样指标。

## 9. 最终里程碑定义

### M1：可飞

- 玩家可稳定飞行
- 有默认巡航、转向、侧倾
- 参数可调

### M2：飞得爽

- 速度感明显
- 高空不再静止
- 手感具备轻重差异

### M3：能对战

- 可追逐、锁定、射击、击毁
- HUD 和战斗规则完整
- 行为日志可采样

### M4：可验收

- PRD 指标基本达成
- 性能与体验都可复现验证
- 决策优先的评估路径可落地
